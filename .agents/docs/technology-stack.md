# Target Technology Stack

This record owns Musubi's selected technologies, the few non-default restrictions needed to use them together, and the conditions that would justify reconsidering a selection. It does not own product behavior, architecture, migration steps, or acceptance procedures.

## Packaging constraints (Void 0.10.10)

- `void` and `@void/vue` are pinned together at `0.10.10`; `vite` is pinned at the verified `8.1.5` public release.
- Keep direct `pathe@2.0.3` until a Void release fixes [void#285](https://github.com/voidzero-dev/void/issues/285). The published prerender runner imports it at runtime but `void@0.10.10` does not declare it, which fails under strict pnpm without the direct dependency.
- Keep the parent-specific pnpm overrides for Void's `@cloudflare/vite-plugin`, `@cloudflare/workers-types`, and Wrangler dependencies aligned with the repository's direct Cloudflare versions. `pnpm peers check` and `vp why` must show one compatible group before changing them.
- Void currently installs `better-sqlite3` through platform features Musubi does not use. `better-sqlite3: false` under `allowBuilds` prevents its install script from running; it is not part of the static client artifact.
- Root `vite.config.ts` owns both the Vite plugins and the Vite+ toolchain configuration. `void prepare` generates `.void/tsconfig.json` and route types before `vue-tsc` checks `src/` through the Void project. A separate NodeNext `tsconfig.tooling.json` lets ordinary `tsc` check `scripts/notion/`, `scripts/font/`, UnoCSS and Vite+ configuration, and their dependencies without adding those tools to the Void source project.
- The known non-fatal Void build and Node warnings are recorded in [Gotchas](./gotchas.md); the static artifact and browser result, not the absence of those messages, determine success.

## Responsibility boundaries

- [Target Architecture](./architecture.md) owns Notion onboarding and schemas, content and settings data flow, Markdown normalization and rendering behavior, static publication, font-delivery behavior, routes, and failure rules.
- [Visual Frontend Development and Acceptance](./visual-frontend-development-and-acceptance.md) owns the required verification process.
- Root `vite.config.ts` under `run.tasks` owns the current `ready` composition and task cache settings. `package.json` owns the leaf scripts those tasks call.
- [DESIGN.md](./DESIGN.md) owns visual tokens and presentation rules.
- Production operations for the maintained example live in [Production Operations](../../docs/production.md).
