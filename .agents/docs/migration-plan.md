# Implementation Migration Plan

## Status and scope

The repository-local migration to the selected architecture is complete. This record retains the completed checklist and concrete acceptance evidence; hosting, deployment triggers, production `hyf.me` content, external Notion writes, and release operations remain outside it. Earlier implementation details remain migration evidence and do not define the target.

## Selected migration constraints

- This migration removes `nuxt-prerender-kit` completely and replaces every accepted responsibility that used it with Nuxt-native server/API routes. A build-only responsibility transfers its data into generated HTML or a page payload and is not emitted as a browser-addressable API.
- Musubi emits no runtime endpoint and requires no running Nitro server. A public static API is added only after a separate concrete browser requirement names it; none is selected now.
- Every accepted dynamic page route selected for prerendering and every API route that must become a public static endpoint is registered explicitly through Nuxt and Nitro prerender mechanisms; link crawling is not treated as a complete route manifest.
- The documented ordinary-user path is fork, duplicate the two-source Notion template, grant a dedicated `Read content` internal integration access to its root, set `NOTION_TOKEN`, `NOTION_CONTENT_DATA_SOURCE_ID`, and `NOTION_CONFIG_DATA_SOURCE_ID`, and generate without source or local configuration edits.
- The canonical public routes are `/`, `/blog/page/:page` beginning at page 2, `/blog/:slug` for Posts, and `/:slug` for Content pages. Tags create no route. Published slugs are explicit and route conflicts fail generation.
- The supplied `musubi.hyf.me` sources are iteration fixtures, not the Yunfei-site compatibility inventory. Do not fabricate absent Yunfei pages during iteration. When the production source is connected, the five audited `hyf.me/blog/:slug` article paths remain unchanged; the broken `/tags` destination is not preserved.

These constraints implement the selected Notion-backed static architecture. They do not select the hosting provider or mechanism that triggers a deployment.

Repository-local implementation revision `d06907f` uses Node `24.18.0`, pnpm `10.24.0`, Vite+ `0.2.4`, Nuxt `4.4.8`, Vue `3.5.39`, Nitro `2.13.4`, and Nuxt's upstream Vite `7.3.6`. The private Notion adapter pins official [`@notionhq/client` `5.23.0`](https://github.com/makenotion/notion-sdk-js/releases/tag/v5.23.0) and the explicit [`2026-03-11` API version](https://developers.notion.com/reference/versioning). The disposable reference used create-nuxt `3.36.1`; [Autonomous Decision Research](./decision-research.md#vite-reference-convention) records the generated conventions and why Musubi removed its Vite replacement. `vp why vite` confirms that Nuxt resolves Vite `7.3.6`, while Vite+'s own bundled command remains outside the Nuxt lifecycle.

`vp install` and `vp run --verbose ready` passed. The `ready` run executed all seven tasks with zero cache hits and cache disabled for every task. Repeated Nuxt type-check and artifact tasks executed again with zero cache hits. The gate prepared 17 Published pages from 24 Content rows and nine Config rows, registered 19 product routes, generated the Nuxt artifact, replaced Nuxt's client-only fallback document with the visible static 404, and verified the final public artifact.

The final `.output/public` contains 29 files and 546,421 bytes. It contains generated HTML, CSS, the pinned Luo font, the generated matching fallback, and font licenses. The fixture has no Published remote image or file, so this artifact contains no stabilized content asset; the HTTPS downloader separately proved content-addressed public download plus rejection of private-network targets. The artifact contains no JavaScript or MJS file, Nuxt payload, public API, build-only route, short-lived Notion URL, symlink, `200.html`, or internal fallback route. A server exposing only this directory returned 200 for `/`, `/about`, `/blog/page/2`, and `/blog/git-aliases`; it returned the visible 404 document with status 404 for an absent page, `/tags`, `/blog/page/1`, and `/api/build/shell`; and it rejected POST with 405.

The fixture's public typography corpus selected U+00B7 and U+2014, both absent from the pinned Luo input and present in the 1,024-byte generated `Musubi CJK Fallback`. A separate uncommitted diagnostic run through the same generator included Luo-covered and fallback-provided Han characters plus Chinese punctuation, English, metadata, headings, prose, quote, and code. `agent-browser 0.31.1` with Headless Chrome `150.0.0.0` loaded both fonts and the final pixels were inspected at 1440×900 and 390×844 in light and warm-dark themes without document overflow; the diagnostic files were removed before the final artifact check.

The same browser inspected the static home, a code-heavy Post, a Content page, page 2 reached from the index through an ordinary full-document link, an X link card, keyboard focus and skip link, system and explicit theme selection, direct entry, hard refresh, narrow layout, and the visible 404. The artifact intentionally emits no client JavaScript, so SPA navigation and hydration are not applicable rather than untested. No unexpected console error, page error, failed request, external X resource, or document-level horizontal overflow remained. The article body was present only in generated HTML, and browser navigation requested neither Notion nor a build-only endpoint.

The final exact-value scan found all three Notion environment values absent from repository files outside ignored private state and absent from `.output/public`. The private build artifact also contains none of those values. An independent final implementation audit found no remaining blocker or major issue after rechecking the pinned-IP HTTPS asset path, size limits, supported Notion syntax, rejected content, formatting, linting, type checking, and static artifact.
